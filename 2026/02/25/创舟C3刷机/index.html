创舟C3刷机

2026-02-25 Comments

a

今天突然发现创舟C3的AV口后面有个隐藏的按钮，之前苦于没有找到短接口，打算试试这个隐藏按钮

测试方案 操作步骤 预期成功标志 如果成功，意味着什么
方案A：测试能否进入FEL模式（线刷） 1. 盒子完全断电。 2. 用双公头USB线连接盒子和电脑（只连USB，不连电源）。 3. 按住这个隐藏按钮不放。 4. 保持按住，给盒子插上12V电源。 5. 观察电脑设备管理器。 电脑出现"未知设备"或"Allwinner USB FEL"。 这就是我们之前苦寻的FEL模式触发按钮！你可以直接用PhoenixSuit进行线刷。
方案B：测试能否从U盘启动（卡启动） 1. 将写好的Armbian U盘/TF卡插入盒子。 2. 盒子完全断电。 3. 按住这个隐藏按钮不放。 4. 保持按住，给盒子插上12V电源。 5. 观察HDMI输出。 屏幕出现Linux启动命令行日志。 这是强制从外部介质启动的按钮！这正是我们上一轮讨论的最理想方案。

方案A可以！这个隐藏按钮确实是可以进入FEL模式，老贼满我好久，现在试试方案B

方案B不可以

接下来进行线刷Armbian

从零到一：H3机顶盒eMMC识别全记录（含指令）

这是一个从“完全黑盒”到“成功驱动”的完整调试历程，每一步都踩在点上。

第一阶段：信息摸底（搞清楚有什么）

步骤 操作 指令 发现
1 查看U-Boot日志 开机时通过HDMI输出 芯片是 H3，内存 1GB，但 eMMC 没出现
2 拆机找芯片 物理查看 找到三星 KLMCG8WEBC-B031，确认是 eMMC
3 尝试多种Armbian镜像 lsblk 检查 5.10/6.x 内核都不认 eMMC，3.4.113 内核也认不出

第二阶段：问题定位（为什么认不出）

步骤 操作 指令 发现
4 检查驱动 `lsmod grep mmc lsmod
5 手动加载驱动 modprobe sunxi-mmc 加载成功，但设备没出现
6 查看内核日志 `dmesg grep -i mmc dmesg
7 查看设备树状态 cat /sys/firmware/devicetree/base/soc/mmc@1c11000/status disabled——mmc2 控制器被禁用
8 查看compatible属性 cat /sys/firmware/devicetree/base/soc/mmc@1c11000/compatible allwinner,sun7i-a20-mmc——设备树把H3的mmc2错误识别成A20的控制器

核心结论：设备树文件里，eMMC 所在的控制器被禁用了，而且 compatible 还配错了。

第三阶段：设备树排查（找出谁是对的）

步骤 操作 指令 发现
9 安装dtc工具 apt update && apt install device-tree-compiler -y （网络问题需要换源）
10 配置国内源 修改 /etc/apt/sources.list 为归档源 解决 apt 404 问题
11 列出所有H3设备树 ls /boot/dtb-*/*h3*.dtb 有 20+ 个 .dtb 文件
12 反编译检查每个dtb `for dtb in /boot/dtb-5.10.34-sunxi/h3.dtb; do echo “=== $(basename $dtb) ===”; dtc -I dtb -O dts $dtb 2>/dev/null grep -A5 “mmc@1c11000”
13 检查当前启动脚本 cat /boot/boot.cmd fdtfile 没指定，用的是默认值

第四阶段：动手修复（手动指定设备树）

步骤 操作 指令 结果
14 备份boot.cmd cp /boot/boot.cmd /boot/boot.cmd.bak-$(date +%Y%m%d-%H%M%S) ✅ 留了后路
15 编辑boot.cmd nano /boot/boot.cmd 找到主线内核部分
16 修改主线内核部分 在 load ${devtype} ${devnum} ${fdt_addr_r} ${prefix}dtb/${fdtfile} 前加上： setenv fdtfile "sun8i-h3-orangepi-pc-plus.dtb" echo "Using device tree: ${fdtfile}" 强制指定PC Plus设备树
17 重新编译boot.scr mkimage -C none -A arm -T script -d /boot/boot.cmd /boot/boot.scr ✅ 成功生成
18 同步并重启 sync && reboot 屏幕上打印 Using device tree: sun8i-h3-orangepi-pc-plus.dtb
19 检查eMMC lsblk mmcblk2 出现了！

🔧 关键转折点

问题 解决方案
mmc2 被禁用 找到 pc-plus.dtb，它把 mmc2 设成了 okay
启动脚本用的默认 dtb 手动在 boot.cmd 里 强制指定 fdtfile
不知道哪个 dtb 能用 用 dtc 反编译逐个检查，找到唯一正确的那个

🧠 经验总结

  1. 设备树是硬件的说明书——不对就点不亮
  2. dmesg 和 lsmod 是最好的医生——先看驱动有没有，再看硬件认不认
  3. 备份是亲爹——改之前先 cp
  4. FEL 模式是全志芯片的最后防线——软件搞不定还有硬件后门
  5. 国内源配置很重要——Debian 10 已 EOL，必须用 archive 路径

第五阶段：转换 boot 分区为 FAT32（解决 Can’t set block device）

时间范围：从 eMMC 识别为 mmcblk2 之后，到最终实现从 eMMC 自动启动。

环境说明：

  • 已在 U 盘系统（通过旧 U-Boot 或 FEL 启动）
  • eMMC 设备 /dev/mmcblk2，原有分区：mmcblk2p1 (boot)、mmcblk2p2 (root)
  • 目标：让 U-Boot 自动从 eMMC 启动，不再依赖 FEL/键盘

5.1 备份当前 boot 分区数据

步骤 操作 完整指令 说明
5.1.1 挂载 boot 分区 mount /dev/mmcblk2p1 /mnt
5.1.2 创建备份目录 mkdir -p ~/boot_backup
5.1.3 复制所有文件到备份目录 cp -rv /mnt/* ~/boot_backup/ cp -rv /mnt/.* ~/boot_backup/ 2>/dev/null 保留隐藏文件，忽略错误
5.1.4 卸载 boot 分区 umount /mnt

5.2 格式化为 FAT32

步骤 操作 完整指令 说明
5.2.1 格式化 boot 分区 mkfs.vfat -F 32 /dev/mmcblk2p1 强制使用 FAT32 格式

5.3 恢复数据到新格式化的分区

步骤 操作 完整指令 说明
5.3.1 重新挂载 boot 分区 mount /dev/mmcblk2p1 /mnt
5.3.2 恢复文件 cp -rv ~/boot_backup/* /mnt/ cp -rv ~/boot_backup/.* /mnt/ 2>/dev/null FAT32 不支持软链接，但内核文件是普通文件，不影响
5.3.3 查看恢复后的文件 ls -la /mnt 确认 vmlinuz-*、uInitrd-*、dtb-* 存在
5.3.4 卸载 boot 分区 umount /mnt

第六阶段：修复环境变量（让 U-Boot 记住配置）

6.1 发现问题：环境变量存到了错误的设备

步骤 操作 完整指令 说明/结果
6.1.1 在 U-Boot 命令行设置启动命令 setenv bootcmd 'mmc dev 1; mmc rescan; fatload mmc 1:1 0x46000000 /vmlinuz-5.10.34-sunxi; fatload mmc 1:1 0x47000000 /dtb-5.10.34-sunxi/sun8i-h3-orangepi-pc-plus.dtb; fatload mmc 1:1 0x48000000 /uInitrd-5.10.34-sunxi; bootz 0x46000000 0x48000000 0x47000000' 当前会话有效
6.1.2 尝试保存环境变量 saveenv 报错：Saving Environment to FAT... MMC: no card present 说明 U-Boot 试图把环境变量存到 SD 卡 (mmc0)
6.1.3 重启后验证 reset 变量丢失，U-Boot 用默认配置

6.2 在 Linux 里用 fw_setenv 写入环境变量

步骤 操作 完整指令 说明/结果
6.2.1 查看 fw_setenv 配置 cat /etc/fw_env.config 显示 /dev/mmcblk0 0x400000 0x20000——指向了错误的设备
6.2.2 备份原配置 cp /etc/fw_env.config /etc/fw_env.config.bak
6.2.3 修改配置文件 nano /etc/fw_env.config 把 /dev/mmcblk0 改成 /dev/mmcblk2
6.2.4 创建新的环境变量文件 cat > ~/uboot-env-clean.txt << 'EOF' stdout=vidconsole stderr=vidconsole bootcmd=mmc dev 1; mmc rescan; fatload mmc 1:1 0x46000000 /vmlinuz-5.10.34-sunxi; fatload mmc 1:1 0x47000000 /dtb-5.10.34-sunxi/sun8i-h3-orangepi-pc-plus.dtb; fatload mmc 1:1 0x48000000 /uInitrd-5.10.34-sunxi; bootz 0x46000000 0x48000000 0x47000000 bootdelay=3 EOF 手动创建干净的环境变量
6.2.5 写入环境变量 fw_setenv -s ~/uboot-env-clean.txt 成功写入
6.2.6 验证写入结果 fw_printenv stdout fw_printenv stderr fw_printenv bootcmd fw_printenv bootdelay 显示正确的值
6.2.7 重启测试 reboot HDMI 无输出——因为环境变量里的 stdout 虽然改成了 vidconsole，但 U-Boot 加载环境变量前已经用了串口

6.3 清空 eMMC 环境变量区域（让 U-Boot 用默认配置）

步骤 操作 完整指令 说明/结果
6.3.1 重新进入 U 盘系统 通过 FEL 或旧 U-Boot 启动 确保能在 Linux 里操作 eMMC
6.3.2 确认 eMMC 设备 lsblk 确认 mmcblk2 存在
6.3.3 清空环境变量区域 dd if=/dev/zero of=/dev/mmcblk2 bs=1M seek=1 count=1 status=progress 清除 1MB 偏移量处的数据（U-Boot 环境变量通常在此）
6.3.4 同步并重启 sync reboot 重启后 U-Boot 找不到环境变量，用默认配置，HDMI 输出恢复

第七阶段：重新构建 eMMC 系统

7.1 重新格式化 eMMC 分区

步骤 操作 完整指令 说明
7.1.1 卸载可能挂载的分区 umount /dev/mmcblk2p1 2>/dev/null umount /dev/mmcblk2p2 2>/dev/null
7.1.2 格式化 boot 分区为 FAT32 mkfs.vfat -F 32 /dev/mmcblk2p1 关键步骤：确保 U-Boot 能用 fatload
7.1.3 格式化根分区为 ext4 mkfs.ext4 /dev/mmcblk2p2

7.2 备份当前 U 盘系统的 boot 文件

步骤 操作 完整指令 说明
7.2.1 创建备份目录 mkdir -p ~/boot_backup
7.2.2 复制 boot 目录所有文件 cp -rv /boot/* ~/boot_backup/ cp -rv /boot/.* ~/boot_backup/ 2>/dev/null

7.3 挂载 eMMC 分区

步骤 操作 完整指令 说明
7.3.1 创建挂载点 mkdir -p /mnt/emmc mkdir -p /mnt/emmc/boot
7.3.2 挂载根分区 mount /dev/mmcblk2p2 /mnt/emmc
7.3.3 挂载 boot 分区 mount /dev/mmcblk2p1 /mnt/emmc/boot

7.4 从 U 盘同步系统到 eMMC

步骤 操作 完整指令 说明
7.4.1 同步根文件系统 rsync -aAXv /* /mnt/emmc --exclude={/dev/*,/proc/*,/sys/*,/tmp/*,/run/*,/mnt/*,/media/*,/lost+found,/boot/*} 保留权限，排除特殊目录
7.4.2 同步 boot 目录 rsync -aAXv /boot/* /mnt/emmc/boot/ 复制内核、initrd、dtb 等

7.5 编写正确的启动脚本

步骤 操作 完整指令 说明
7.5.1 进入 eMMC boot 目录 cd /mnt/emmc/boot
7.5.2 创建新的 boot.cmd cat > boot.cmd << 'EOF' setenv bootargs console=ttyS0,115200 console=tty0 root=/dev/mmcblk2p2 rootwait mmc dev 1 fatload mmc 1:1 0x46000000 /vmlinuz-5.10.34-sunxi fatload mmc 1:1 0x47000000 /dtb-5.10.34-sunxi/sun8i-h3-orangepi-pc-plus.dtb fatload mmc 1:1 0x48000000 /uInitrd-5.10.34-sunxi bootz 0x46000000 0x48000000 0x47000000 EOF 手动编写，不依赖 Armbian 模板
7.5.3 删除旧的 boot.scr rm -f boot.scr 确保旧脚本被清除
7.5.4 生成新的 boot.scr mkimage -C none -A arm -T script -d boot.cmd boot.scr 生成 U-Boot 可执行的脚本
7.5.5 查看生成的文件 ls -l boot.scr 大小应为 300-400 字节
7.5.6 返回根目录 cd /

7.6 重新写入永久的 U-Boot 环境变量

步骤 操作 完整指令 说明
7.6.1 确认 fw_env.config 正确 cat /etc/fw_env.config 应指向 /dev/mmcblk2
7.6.2 创建最终的环境变量文件 cat > ~/uboot-env-final.txt << 'EOF' stdout=vidconsole stderr=vidconsole bootcmd=mmc dev 1; mmc rescan; fatload mmc 1:1 0x46000000 /vmlinuz-5.10.34-sunxi; fatload mmc 1:1 0x47000000 /dtb-5.10.34-sunxi/sun8i-h3-orangepi-pc-plus.dtb; fatload mmc 1:1 0x48000000 /uInitrd-5.10.34-sunxi; bootz 0x46000000 0x48000000 0x47000000 bootdelay=3 EOF
7.6.3 写入环境变量 fw_setenv -s ~/uboot-env-final.txt
7.6.4 验证写入结果 fw_printenv stdout fw_printenv stderr fw_printenv bootcmd fw_printenv bootdelay 所有值应正确显示

7.7 卸载并重启

步骤 操作 完整指令 说明
7.7.1 卸载 boot 分区 umount /mnt/emmc/boot
7.7.2 卸载根分区 umount /mnt/emmc
7.7.3 同步并重启 sync reboot
